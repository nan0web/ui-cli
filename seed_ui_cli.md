# Мета: Імплементація Interactive Content Viewer та Inline Forms у @nan0web/ui-cli

**Контекст:**
Створюється новий підхід Data-Driven UI (через `App-as-a-Model` і Markdown `$content`). Оскільки CLI не має DOM-бровзера для рендерингу HTML, нам потрібен власний інтерактивний Markdown/HTML Viewer компонент (контрол), який буде рендерити контент з YAML-AST, а також підтримувати інтерактивну навігацію та вбудовані форми без виходу з поточного "вікна".

## Завдання для Агента:

### 1. Новий компонент: `ContentViewer` (Interactive Layout)
Створити новий UI-контрол (наприклад, `ContentViewer` / `MarkdownViewer`), який приймає `content` масив (AST нашої сторінки) і відображає його як скрол-об'єкт на весь екран (з урахуванням лімітів `process.stdout.rows`).

**Вимоги до `ContentViewer`:**
* **Left-Gutter (Ліва колонка)**: Залишати 3–5 символів зліва для індикаторів форматування (напр. `1 =` або `H1 |` для заголовків, `---` для HR, `p  |` для параграфів).
* **Скролинг**: Підтримка навігації клавішами **⬆ (Вгору)** та **⬇ (Вниз)**. Посторінковий скрол або порядковий. Якщо довгого тексту багато — автоматичний Word-Wrap по ширині терміналу.
* **Footer & Гарячі клавіші**: Внизу екрану стабільно виводити Footer-рядок, який містить:
  - Відсоток прочитаного `[ 45% ]`
  - Кількість фокусних елементів поблизу
  - Гарячі клавіші підказок: `[Enter: обрати] [Esc: назад] [⬆/⬇: скрол] [⬅/➡: фокус]`

### 2. Фокусна навігація всередині тексту (Links & Inline Actions)
Компонент має розрізняти "пасивний" текст і "інтерактивний" контент (посилання, кнопки, розгорнуті форми) всередині блоку (наприклад, масив елементів у параграфі).
* Натискання клавіш **⬅ (Вліво)** / **➡ (Вправо)** (або Tab) має перемикати фокус (підсвічувати) між видимими на екрані інтерактивними елементами.
* **Enter (Клік)** по сфокусованому посиланню (`$href`) має завершувати роботу `ContentViewer` і повертати `UI Event` роутеру (наприклад, `action: 'navigate', url: '...'`) для переходу на нову сторінку — як у браузері.

### 3. Inline Forms (Згорнуті форми) та Sub-Models
Опрацювання структури типу `App.Feedback.Form: true \n $collapsed: true` (екземпляри інших Моделей всередині сторінки):
* За замовчуванням (якщо $collapsed: true), форма рендериться інтегровано всередині сторінки як список її полів та поточних значень у вигляді одного рядка або блоку (напр., `Поле1: [значення], Поле2: [значення]`).
* Користувач може доскролити до поля, перевести на нього фокус і натиснути **Enter**.
* У цей момент поточний `ContentViewer` може "призупинитися" (очистити екран), щоб запустити повноцінний генератор конкретної форми/поля (Ask Intent). Після завершення введення або натискання **Esc** — управління повертається назад у `ContentViewer`, продовжуючи скрол з того ж самого місця (збереження стану).
* Для зручності дебагу після закриття форми/форми можна залишати в логах терміналу trace-коментарі (що саме відбулося).

**Очікуваний результат:**
Повноцінний TDD-цикл (зокрема через `cli/nan0cli.story.js` чи нові тести) для перевірки `ContentViewer`. Агент має переконатися, що обробка закриття через клавішу Escape коректно хендлиться як вихід на рівень вище або як закриття Inline форми.
